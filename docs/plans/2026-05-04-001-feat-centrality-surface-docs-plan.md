# Centrality Surface — Tool Reference Docs Followup

## Overview

Close out the last open unit (U6 — Documentation) from
`docs/plans/2026-04-23-007-feat-centrality-surface-plan.md`. Centrality computation, caching,
query exposure, and tests all shipped in `c5d3e17`, and `CLAUDE.md` already references the new
field. The remaining gap is the per-family tool reference under
`plugins/kitty/skills/kitty/references/tool-reference/`, which still describes the response
shapes for `query_node`, `search`, `rank_nodes`, `get_context_summary`, and
`get_file_structure` without mentioning the new `centrality: float` field. Because that
directory lives inside the `Kakise/cartographing-kitties-skills` Git submodule, the change
cannot land as a direct edit in this repo — it must go through a submodule PR.

## Problem Frame

Predecessor `docs/plans/2026-04-23-007-feat-centrality-surface-plan.md` exposed a weighted-
PageRank `centrality` score in `[0, 1]` on every node summary returned by `_summarise_node`.
Five MCP tool families surface the field today: `query_node`, `search`, `rank_nodes`,
`get_context_summary`, and `get_file_structure`. The autogenerated tool reference
(generated by `scripts/generate_tool_reference.py` from FastMCP introspection) still emits
descriptions that pre-date U4, so agents reading the reference do not learn that they can
sort or rank by `centrality` directly. The reference is the canonical place callers look
for response-shape documentation, so leaving it stale defeats the point of shipping the
field.

## Memory Context

Treat-box entry #2 (convention) reaffirms that per-unit `**State:**` lines must be
standalone paragraphs under each header — the audit silently skips checkbox-bulleted state
lines. This plan follows that contract.

A second, plan-specific lesson: skill content lives in a Git submodule. Direct edits to
`plugins/kitty/skills/kitty/references/tool-reference/*.md` from this repo would not be
upstreamable. The correct workflow is a PR against
[`Kakise/cartographing-kitties-skills`](https://github.com/Kakise/cartographing-kitties-skills),
followed by a submodule pointer bump in this repo.

## Implementation Units

### Unit 1 — Tool reference centrality field documentation

**State:** in_progress

- [ ] **Goal.** Document the `centrality: float` field on every tool response that emits it
      (`query_node`, `batch_query_nodes`, `search`, `rank_nodes`, `get_context_summary`,
      `get_file_structure`) in the per-family tool reference, and land the change via a
      submodule PR.
- [ ] **Files (in the skills submodule, not this repo).**
  - **Modify:**
    - `plugins/kitty/skills/kitty/references/tool-reference/query.md` — extend the response
      schema sections for `query_node`, `batch_query_nodes`, `search`,
      `get_context_summary`, and `get_file_structure` to call out `centrality` (range,
      semantics, freshness contract).
    - `plugins/kitty/skills/kitty/references/tool-reference/analysis.md` — extend the
      response schema section for `rank_nodes` likewise.
- [ ] **Approach.**
  - Add a one-paragraph framing block at the top of each modified file (or in a shared
    "Centrality" subsection) noting that `centrality` is a weighted-PageRank score in
    `[0, 1]`, refreshed lazily after each `graph_version` bump, and safe to use as a
    stable secondary sort key.
  - Per-tool, append a `centrality` row to the response-field table, citing the
    `_summarise_node` definition in `src/cartograph/server/tools/query.py` for the canonical
    value.
  - If the per-family reference is autogenerated by `scripts/generate_tool_reference.py`,
    confirm whether the description flows from the FastMCP tool docstrings or the jinja2
    template. If the docstrings are the source of truth, the doc fix lands here in this
    repo (in `src/cartograph/server/tools/query.py` and `analysis.py`) and the submodule
    PR only re-runs the generator. If the jinja2 template is the source of truth, the
    submodule PR edits the template and rebuilds.
- [ ] **Submodule PR workflow.**
  1. From `plugins/kitty/skills/`, branch off `main` of the submodule:
     `git checkout -b feat/document-centrality-field`.
  2. Edit `kitty/references/tool-reference/query.md` and `analysis.md` (or the shared
     template / source docstring, if generated).
  3. Commit, push, and open a PR against `Kakise/cartographing-kitties-skills`. Reference
     this plan and predecessor `2026-04-23-007` in the description.
  4. After merge, in this repo bump the submodule pointer:
     `git -C plugins/kitty/skills pull origin main` then `git add plugins/kitty/skills`
     and commit as a chore.
  5. If the docstrings drove the change, also run
     `uv run python scripts/generate_tool_reference.py` in the submodule clone to refresh
     the generated files before the PR.
- [ ] **Test scenarios.**
  - Happy: a fresh checkout of the submodule branch shows `centrality` documented on each
    of the five tools listed above.
  - Edge: running `scripts/generate_tool_reference.py --check` (if applicable) produces no
    diff after the PR merges.
- [ ] **Verification.** Submodule PR is merged and the pointer bump in this repo lands. The
      tool reference now documents `centrality` for every tool that emits it. Plan U6 of
      `2026-04-23-007` is satisfied; that plan can flip to `complete` with an
      `implemented_in:` reference once the submodule pointer commit is recorded here.

## Risks & Dependencies

| Risk | Mitigation |
|---|---|
| Submodule PR review latency blocks closing the predecessor | Plan stays `active` until merged; no downstream plan depends on this doc fix |
| Generated vs. hand-authored doc source ambiguity | First step of Unit 1 confirms which is canonical (`scripts/generate_tool_reference.py` introspection vs. template) |
| Submodule pointer bump forgotten after merge | Mention the bump explicitly in the PR description and the merge checklist |

**Dependencies:** none beyond access to the submodule repo. No code changes in this repo
unless the FastMCP docstrings turn out to be the canonical source.
